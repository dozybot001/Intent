# Intent Documentation I18n Plan

Purpose: define how Intent documentation should evolve into a bilingual EN/CN documentation set.

This document answers:

- how bilingual files should be organized
- how EN/CN naming should work
- how sync should work across EN/CN files
- which files should move first

This document does not answer:

- CLI contract details
- implementation roadmap details
- release or community operations

## Goals

- keep core documents available in both EN and CN
- avoid bilingual drift
- keep maintenance cost bounded

## Current State

- the repository is in the middle of a bilingual migration
- GitHub entry points now surface the English README first
- the public entry-point docs now exist as EN/CN pairs
- `cli` and `roadmap` also now exist as EN/CN pairs

## Core Policy

- use symmetric EN/CN naming
- keep mixed-language sections out of a single file
- use `README.md` as the English GitHub entry and `README.CN.md` as the Chinese root version

## Directory Strategy

Target structure:

```text
README.md
README.CN.md

docs/
  README.md
  EN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
  CN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
```

Rules:

- `README.md` is the English GitHub landing file
- `docs/README.md` is a lightweight language chooser
- documentation lives under `docs/EN/` and `docs/CN/`
- filenames inside each language tree stay consistent
- the root README remains the only naming exception: English uses `README.md`, while Chinese uses `README.CN.md`

## Naming Rules

- use `docs/EN/` for English docs
- use `docs/CN/` for Chinese docs
- keep filenames identical across the two language trees

Examples:

- `docs/EN/vision.md` <-> `docs/CN/vision.md`
- `docs/EN/cli.md` <-> `docs/CN/cli.md`
- `README.md` <-> `README.CN.md`

## Sync Rules

- each paired document should link to its sibling near the top
- when one language lands first, the paired version should be synced deliberately rather than silently drifting
- when helpful, paired docs can note `Last synced with: <commit>`

## Linking Rules

- `README.md` should link to `README.CN.md`
- English docs should link to English docs by default
- Chinese docs should link to Chinese docs by default
- the top language switch link is the place where cross-language navigation happens
- doc indexes should expose both language paths clearly

## Writing Rules

- do not translate CLI commands, object ids, flags, or JSON field names
- keep core object terms in English: `intent`, `checkpoint`, `adoption`, `decision`, `run`
- rewrite for natural English instead of translating line by line
- use clear open-source documentation tone in both languages
- avoid bilingual paragraphs in the same document

## Migration Order

Phase 1: public entry points

- `README.md`
- `README.CN.md`
- `docs/EN/README.md`
- `docs/CN/README.md`
- `docs/EN/vision.md`
- `docs/CN/vision.md`
- `docs/EN/demo.md`
- `docs/CN/demo.md`
- `docs/EN/glossary.md`
- `docs/CN/glossary.md`

Phase 2: project contracts

- `docs/EN/cli.md`
- `docs/CN/cli.md`
- `docs/EN/roadmap.md`
- `docs/CN/roadmap.md`

Phase 3: finish paired coverage

- `docs/EN/i18n.md`
- `docs/CN/i18n.md`
- any later docs should be added as EN/CN pairs

## Recommended Execution Plan

Step 1:

- keep `README.md` as the English GitHub landing file
- maintain `README.CN.md` as the Chinese root version

Step 2:

- move public docs into `docs/EN/` and `docs/CN/`
- update indexes and cross-links to the language-tree structure

Step 3:

- add EN versions for Chinese-only specs such as `cli` and `roadmap`
- keep links explicit while the file pairs are incomplete

Step 4:

- verify that GitHub entry points show English first
- verify that paired language versions remain easy to discover

## Phase Exit Criteria

Phase 1 is complete when:

- `README.md` is the English root README
- `README.CN.md` exists
- public entry-point docs are available as EN/CN pairs

Phase 2 is complete when:

- `docs/EN/cli.md` / `docs/CN/cli.md` both exist
- `docs/EN/roadmap.md` / `docs/CN/roadmap.md` both exist
- major doc links no longer depend on the old suffix-less filenames

Phase 3 is complete when:

- all core docs exist as EN/CN pairs
- sync notes and paired links are consistent across the doc set

## Maintenance Policy

- new public docs should be created as EN/CN pairs whenever practical
- if one language lands first, the other should be queued explicitly

## Short Conclusion

Intent should keep `README.md` / `README.CN.md` at the root, and organize `docs/` into mirrored `EN/` and `CN/` language trees.
