# Intent Documentation I18n Guide

Purpose: define the current bilingual EN/CN documentation structure and the maintenance rules around it.

This document answers:

- how bilingual files should be organized
- how EN/CN naming should work
- how sync should work across EN/CN files
- how the documentation layout should be maintained over time

This document does not answer:

- CLI contract details
- implementation roadmap details
- release or community operations

## Goals

- keep core documents available in both EN and CN
- avoid bilingual drift
- keep maintenance cost bounded

## Current State

- the repository uses a bilingual EN/CN documentation layout
- GitHub shows the English README first
- core docs are available under both `docs/EN/` and `docs/CN/`
- the root README structure is fixed as `README.md` and `README.CN.md`

## Core Policy

- use symmetric EN/CN structure
- keep mixed-language sections out of a single file
- use `README.md` as the English GitHub entry and `README.CN.md` as the Chinese root version

## Directory Strategy

Current structure:

```text
README.md
README.CN.md

docs/
  README.md
  EN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
  CN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
```

Rules:

- `README.md` is the English GitHub landing file
- `docs/README.md` is a lightweight language chooser
- documentation lives under `docs/EN/` and `docs/CN/`
- filenames inside each language tree stay consistent
- the root README remains the only naming exception: English uses `README.md`, while Chinese uses `README.CN.md`

## Naming Rules

- use `docs/EN/` for English docs
- use `docs/CN/` for Chinese docs
- keep filenames identical across the two language trees

Examples:

- `docs/EN/vision.md` <-> `docs/CN/vision.md`
- `docs/EN/cli.md` <-> `docs/CN/cli.md`
- `README.md` <-> `README.CN.md`

## Sync Rules

- each paired document should link to its sibling near the top
- update the current language version first, then sync the paired language version deliberately
- when helpful, paired docs can note `Last synced with: <commit>`

## Linking Rules

- `README.md` should link to `README.CN.md`
- English docs should link to English docs by default
- Chinese docs should link to Chinese docs by default
- the top language switch link is the place where cross-language navigation happens
- `docs/README.md` should remain the language chooser for the `docs/` directory

## Writing Rules

- do not translate CLI commands, object ids, flags, or JSON field names
- keep core object terms in English: `intent`, `checkpoint`
- rewrite for natural English instead of translating line by line
- use clear open-source documentation tone in both languages
- avoid bilingual paragraphs in the same document

## Current Coverage

Core documents currently covered in both language trees:

- `README.md` / `README.CN.md`
- `docs/EN/README.md` / `docs/CN/README.md`
- `docs/EN/glossary.md` / `docs/CN/glossary.md`
- `docs/EN/vision.md` / `docs/CN/vision.md`
- `docs/EN/cli.md` / `docs/CN/cli.md`
- `docs/EN/demo.md` / `docs/CN/demo.md`
- `docs/EN/roadmap.md` / `docs/CN/roadmap.md`
- `docs/EN/i18n.md` / `docs/CN/i18n.md`

## Maintenance Workflow

- update the current language version first
- sync the paired language version deliberately
- keep same-language links inside the same language tree
- keep cross-language navigation only in the top language switch and index pages
- verify that `docs/README.md` still works as the language chooser after structural edits

## Maintenance Policy

- new public docs should be created in both language trees whenever practical
- if one language lands first, the other should be queued explicitly

## Short Conclusion

Intent keeps `README.md` / `README.CN.md` at the root, and organizes `docs/` into mirrored `EN/` and `CN/` language trees with deliberate cross-language sync.
