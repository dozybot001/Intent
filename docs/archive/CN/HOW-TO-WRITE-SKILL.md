# 一、好的 Agent Skill，本质上应该是什么

我会把一个好的 Skill 定义为：

> **在某一类任务上，把“目标—判断—动作—校验—产出”固化下来，使 Agent 在复杂环境中仍能稳定地产出可接受结果。**

所以 Skill 不是：

* 单纯的 prompt 模板
* 一组工具调用说明
* 一堆经验性 tips
* 一段人类读得懂但 Agent 执行不稳的“原则宣言”

Skill 应该至少同时承担 5 个作用：

### 1. 定义任务边界

告诉 Agent：

* 这类任务到底要解决什么
* 什么不属于这个 Skill 的职责
* 什么时候该退出，交给别的 Skill

### 2. 提供世界模型

告诉 Agent：

* 在这类任务里，什么信息重要
* 信息之间是什么关系
* 常见歧义和错误是什么

对于“语义信息采集”，这部分尤其重要。
因为 Agent 失败，往往不是不会搜，而是**不知道该采什么、为什么采、采到哪一层算够**。

### 3. 提供执行流程

不是泛泛说“先理解需求，再收集信息”，而是要明确：

* 输入解析
* 信息槽位拆解
* 采样优先级
* 证据合并
* 冲突处理
* 输出压缩

### 4. 提供质量控制

告诉 Agent：

* 什么叫“已经足够好”
* 什么叫“证据不足”
* 什么叫“高风险猜测”
* 什么情况下必须显式保留不确定性

### 5. 约束输出接口

Skill 最终要产出什么结构：

* summary
* facts
* uncertainties
* evidence map
* missing info
* recommended next action

没有输出契约，Skill 很难复用。

# 二、为什么语义信息采集类 Skill 会天然很大

因为它不是工具问题，而是**认知问题**。

一个弱 Skill 会写成这样：

> 读取用户需求 → 搜索相关内容 → 提取关键信息 → 总结输出

这几乎没用。因为这里面每一步都隐藏了大量判断。

真正复杂的地方在于：

### 1. 任务目标往往是隐式的

用户说“帮我了解这个项目”，实际可能要的是：

* 技术可行性判断
* 商业潜力判断
* 文档质量判断
* 是否值得自己投入时间

Skill 必须教 Agent 做**意图解包**。

### 2. “相关信息”不是天然存在的

Agent 必须知道信息维度，例如：

* 定义信息
* 状态信息
* 约束信息
* 证据信息
* 反例信息
* 风险信息
* 缺失信息

### 3. 语义采集不是“越多越好”

而是要处理：

* 信息过载
* 重复证据
* 来源冲突
* 旧信息污染
* 表述相似但语义不同
* 局部强信号误导整体判断

### 4. 采集和理解是耦合的

Agent 不是先完整收集，再统一理解。
而是应该：

* 先形成假设
* 再定向采样
* 再修正假设
* 再补证据

这本质上是一个**迭代推理过程**。

所以你感觉 Skill 会很大，是因为你已经意识到：
**真正的 Skill 不是“操作手册”，而是“任务领域内的小型认知系统”。**

这个方向是对的。

---

# 三、但 Skill 不能只是“大”，而要“分层”

这是最关键的设计原则。

如果你把所有理解性内容都塞进一个大 prompt，结果通常会变成：

* 可读性差
* 复用性差
* 维护困难
* Agent 抓不住重点
* 不同任务之间互相污染

更好的做法是把 Skill 分成 4 层。

---

## 第 1 层：哲学层 / 任务原则层

回答“这类任务最根本的成功标准是什么”。

例如语义信息采集 Skill 的原则可以是：

* 先理解决策需求，再决定采集范围
* 优先采集能改变判断的信息，而不是容易采集的信息
* 区分事实、解释、推测、建议
* 不以单一来源构建强结论
* 在不确定时保留不确定，而不是补脑

这一层像 constitution。

---

## 第 2 层：领域模型层

回答“这类任务里，世界应该怎样被拆解”。

例如语义信息采集可以拆成这些语义槽位：

* 主体是什么
* 主体的目标是什么
* 当前状态是什么
* 已知约束是什么
* 关键行动者有哪些
* 已有证据有哪些
* 缺失证据有哪些
* 潜在冲突点有哪些
* 对最终决策最敏感的变量是什么

这层非常重要，因为它把“理解性内容”结构化了。
否则 Agent 只会泛泛搜集。

---

## 第 3 层：执行协议层

回答“具体该怎么做”。

例如：

### 输入解析协议

* 识别显式目标
* 推断隐式目标
* 提取任务对象
* 判断时间敏感性
* 判断信息可信度要求

### 信息采样协议

* 先采定义性信息
* 再采状态性信息
* 再采反证和限制条件
* 若发现高不确定点，则展开定向采样
* 若信息已收敛，则停止扩展

### 证据组织协议

* 每条结论必须挂靠证据
* 证据按来源和时间排序
* 冲突证据不能直接平均，应标记冲突
* 旧信息若与新信息矛盾，优先保留新证据但注明变化

### 终止协议

* 核心问题已有足够证据支持
* 新采样边际收益下降
* 剩余不确定性不影响当前决策
* 或者必须明确告诉用户还缺什么

这一层才是 Agent 真正“会做事”的地方。

---

## 第 4 层：输出契约层

回答“最后长什么样”。

例如规定输出必须包含：

* 任务理解
* 已确认事实
* 初步判断
* 关键证据
* 主要不确定点
* 建议下一步采集方向

这样 Skill 的结果才能接到下游系统里。

---

# 四、一个好的 Skill，不是把所有知识写进去，而是把“判断点”写出来

这是一个很重要的设计观。

很多人写 Skill 时会想：

> 我要把所有可能遇到的情况都写全。

结果会导致 Skill 爆炸。

更好的思路是：

> **不用穷举所有内容，但要穷举关键判断点。**

例如对于语义采集，不需要把所有行业知识塞进去，但需要写清楚 Agent 必须判断：

* 当前任务的决策对象是什么
* 哪些信息会影响决策
* 哪些信息只是背景噪声
* 证据是否独立
* 是否存在时间过期
* 是否存在定义漂移
* 是否存在来源偏见
* 是否需要继续采样

所以 Skill 的核心不是“知识总量”，而是“决策骨架”。

# 五、我建议你把 Skill 看成“三部分的组合”

## A. Instruction

告诉 Agent 原则与流程。

## B. Schema

告诉 Agent 如何表示它看到的世界。

比如你可以给一个固定中间结构：

```yaml
task:
  objective:
  hidden_objective:
  scope:
entity_map:
  primary_entities:
  related_entities:
information_slots:
  definitions:
  states:
  constraints:
  evidence:
  counterevidence:
  unknowns:
risk_flags:
  ambiguity:
  stale_info:
  source_conflict:
  hallucination_risk:
decision_relevance:
  high_impact_factors:
  low_value_info:
output_plan:
  summary:
  evidence_backed_claims:
  unresolved_questions:
```

这会极大提高稳定性。

## C. Exemplars

给少量高质量例子，教 Agent：

* 什么是好的采集
* 什么是坏的采集
* 什么叫过采样
* 什么叫漏采关键变量

很多理解性内容，其实最好通过 exemplar 注入，而不是长篇原则灌输。

# 六、语义信息采集 Skill 的核心难点，不在“采”，在“选择”

真正高级的 Skill，必须教 Agent 这几件事：

### 1. 采什么

围绕决策敏感变量采，而不是围绕表面关键词采。

### 2. 不采什么

主动忽略不能改变结论的信息。

### 3. 采到什么程度停止

不是“越多越好”，而是“足以支持当前判断即可”。

### 4. 怎么面对不完整信息

允许输出：

* 当前最稳妥判断
* 证据覆盖范围
* 还缺哪块证据

### 5. 怎么把信息变成结构化理解

不是摘抄，不是 summarization，而是**形成可行动的状态表征**。

# 七、好的 Skill 应该长什么样：一个实用模板

下面我给你一个适合你这种 Agent/Intent/Skill 方向的模板骨架。

---

## 1. Skill Identity

* 名称
* 适用任务
* 不适用任务
* 输入前提
* 输出承诺

## 2. Success Criteria

* 成功的定义
* 最低可接受结果
* 失败信号

## 3. Core Mental Model

* 本任务中最关键的对象是什么
* 信息如何分类
* 决策如何被信息影响
* 常见误判来源

## 4. Operating Procedure

* Step 1: Clarify objective
* Step 2: Build semantic frame
* Step 3: Identify critical unknowns
* Step 4: Prioritize evidence gathering
* Step 5: Reconcile conflicting signals
* Step 6: Produce evidence-backed summary

## 5. Heuristics

* 何时扩展搜索
* 何时停止
* 何时保守回答
* 何时要求补充信息
* 何时切换到别的 Skill

## 6. Output Schema

* summary
* evidence
* uncertainties
* next actions

## 7. Failure Modes

* 表面相关性误判
* 关键词绑架
* 信息冗余
* 忽视反证
* 把推测说成事实
* 旧信息污染新判断

## 8. Examples

* 好例子
* 坏例子
* 对比例子

# 八、一个很重要的结论：Skill 大，不代表 prompt 长

这点非常关键。

你感觉需要注入很多理解性内容，这没有错。
但它不一定要以“超长自然语言 prompt”的形式存在。

更理想的形态是：

* **短核心指令**：定义原则
* **结构化 schema**：定义中间表示
* **流程协议**：定义执行顺序
* **案例集**：定义质量边界
* **可组合子 skill**：处理局部任务

也就是说：

> **Skill 的“知识体积”可以很大，但运行时激活的上下文应该尽量局部化。**

这和软件工程很像：

* 不是写一个 5000 行函数
* 而是写一个系统，由多个模块组成

# 九、对于你做 Agent Skill，我给你一个更强的设计判断

你应该避免把 Skill 设计成：

> “告诉 Agent 应该怎么想”

而应该设计成：

> “规定 Agent 在哪些节点必须做出哪些判断，并以什么格式记录这些判断”

因为前者太玄，后者才能工程化。

比如不要只写：

* “注意理解用户真实意图”
* “注意信息可靠性”
* “注意避免遗漏重要信息”

而要写成：

* 在开始采集前，必须显式写出 primary objective 和 hidden objective
* 每个核心结论都必须关联至少一类 supporting evidence
* 每轮采集后都要更新 unknowns 和 risk flags
* 若 source conflict 存在，禁止输出强确定表述

这就从“提示词”进入“协议”了。

# 十、我对你的问题的直接回答

你的直觉是对的：

### 一个好的 Agent Skill 设计，应该具备这些特征：

* 有明确任务边界
* 有明确成功标准
* 有任务专属的语义世界模型
* 有可执行的流程协议
* 有中间状态表示
* 有质量控制与停止条件
* 有稳定输出接口
* 有失败模式说明
* 有高质量 exemplars

### 对于语义信息采集类 Skill：

* Skill 往往确实会比较大
* 需要大量理解性内容
* 但不能只是长篇说明
* 必须把理解性内容压成结构、协议、槽位、启发式和案例

### 最核心的一句话：

> **好的 Skill 不是“信息很多”，而是“让 Agent 在关键判断点上不犯低级错”。**
