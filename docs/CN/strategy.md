# Intent 发展战略

这份文档定义从当前本地 CLI 到成为新范式的路径。它不是功能路线图——是战略方向。

## Intent 补的是什么缺口

Git 记录代码怎么变了，但不记录为什么。

在人类时代，这个缺口可以靠记忆、靠讨论、靠"问一下当时写代码的人"来弥补。在 agent 时代，这些补偿机制全部失效：

- agent 没有持久记忆
- agent 不能"问一下上次那个 agent"
- 每次新 session 都从零开始理解上下文

这意味着 Intent 不是在解决一个"有了更好"的问题，而是在解决一个"没有就不行"的问题——只是这个"不行"会随着 agent 驱动开发成为主流而变得不可忽视。

## 第一阶段：Agent 记忆层

**当前最疼的点不是"记录历史"，而是"恢复上下文"。**

每一个用过 Claude Code、Codex、Cursor 的人都经历过：上一轮对话做了一半的工作，新开一轮对话时 agent 完全不知道之前做了什么、为什么做到一半停了、哪些路径已经试过了。

这就是 Intent 的第一个杀手场景：**跨 session 上下文恢复**。

具体做法：

- 让 `itt inspect` 的输出直接成为 agent 启动上下文的一部分
- 写一个极轻的 hook：agent 启动时自动读 `inspect`，结束时自动 `snap` 或 `done`
- 目标体验：用户开一个新 session，agent 开口就说"上次你在做 X，做到了 Y 步骤，停下来的原因是 Z，我建议继续从这里开始"

用户不需要理解 Intent 是什么。用户只会觉得"这个 agent 怎么知道我上次做到哪了"。**最好的基础设施是用户感知不到的基础设施。**

## 第二阶段：语义交换协议

Git 成为范式不是因为 Linus 写了一个好用的 CLI，而是因为 GitHub 把 Git 变成了社交基础设施。

Intent 的 GitHub 等价物是 IntHub。但现阶段更务实的路径是：**成为 agent 平台之间的语义交换协议。**

当前的问题：Claude Code 做了一半的工作，交给 Codex 继续，没有任何标准化的方式传递"做到哪了、为什么"。每个平台都在发明自己的上下文管理。

Intent 可以成为这个中间层：

- 把 `.intent/` 定义为一个极简的开放规范（已经有了）
- 为主流 agent 平台写原生集成（不是 skill 文件，是真正的 plugin）
- 价值锚点从 CLI 工具转向 **`.intent/` 目录本身就是协议**

当开发者开始讨论是否把 `.intent/` 加入 `.gitignore` 还是 commit 进仓库时，Intent 就已经成为基础设施了。

## 第三阶段：语义历史的网络效应

当足够多的仓库里有 `.intent/` 目录时，新的可能性会出现：

- **Intent-aware code review**：PR 不再只是 diff，而是"这个 PR 解决了什么 intent，经历了哪些步骤，关键的 rationale 是什么"
- **团队级语义仪表盘**：不是看谁提交了多少行代码，而是看团队正在解决什么问题、卡在哪里
- **AI 辅助的决策考古**：几个月后回来看代码，不用靠 git blame 猜，直接查"当时为什么选了这个方案"

这就是 IntHub 真正该做的事。但它必须建立在第一和第二阶段的基础上。

## 什么会杀死 Intent

1. **agent 平台自己内建了上下文管理。** 如果 Claude Code 原生支持跨 session 记忆，Intent 的第一个场景就没了。但目前来看，各平台更倾向于做通用记忆，而不是结构化的语义历史——这是不同的东西。

2. **过早平台化。** 在没有证明本地价值之前就去做 IntHub、做协作、做 UI，会把精力分散到错误的地方。

3. **变成"又一个 dev tool"。** 如果 Intent 的叙事是"一个帮你管理开发过程的工具"，它会淹没在几百个类似的工具里。Intent 的叙事必须是：**Git 记录代码怎么变了，Intent 记录为什么这样变。两层合在一起才是完整的工程历史。**

## 当前下一步

1. 写一个 Claude Code hook，让 Intent 在 session 开始和结束时自动工作，零手动操作
2. 制造一个 demo：第一个 session 做一半工作，第二个 session 自动接上，用户什么都不用解释
3. 把 `.intent/` 提交进一两个真实开源项目，让其他开发者和 agent 在真实仓库中遇到它

范式不是设计出来的，是长出来的。但种子要种在对的地方。
