# 首个 Agent 试用反馈

这份文档记录了第一个在帮助开发 Intent 本身时使用 Intent 的 agent 的实际反馈。

它不是 CLI contract 文档，而是把真实使用中得到的产品反馈沉淀下来，尤其面向长链路 agent 工作流。

## 当前判断

Intent 已经有帮助了，尤其是在任务链路变长之后。

它最核心的价值，不是让每一步都更快，而是让 agent 在任务不断变形的过程中，不容易丢掉语义上下文。

目前 Intent 已经明显帮助到这些事情：

- 让长任务始终挂在一条稳定的语义主线上
- 把候选方案和最终采纳分开
- 不只保留“改了什么”，也保留“为什么这么改”
- 通过 `itt inspect --json` 给 agent 一个相对稳定的机器读取界面

## 已经有效的地方

### 1. 任务越长，Intent 的价值越明显

在短任务或纯只读交互里，Intent 有帮助，但不是必需品。

一旦进入多步实现、设计修正、安装路径调整、文档口径收敛和 release 准备，Intent 的价值就会清楚很多。它能明显减少上下文漂移，也更容易在中途转向后重新接上。

### 2. 这层对象模型比纯 Git 历史更适合表达语义过程

Git 当然仍然是代码历史的来源，但它不能很好地区分：

- 一个候选想法
- 一个被采纳的方向
- 一个“为什么选它”的决策

Intent 在这件事上补位是有效的。

### 3. `inspect --json` 很适合继续成为 agent 的主读取入口

agent 在做下一次写操作前，需要一个稳定可信的读取命令。

`itt inspect --json` 现在已经很接近这个角色了，后续应该继续把它打磨成 agent 工作流里的主状态读取面。

## 主要摩擦

### 1. 状态机仍然比 happy path 所需要的更严格

当使用保持在主路径上时，Intent 是连贯的。

但只要稍微离开 happy path，CLI 还是会显得偏硬。`STATE_CONFLICT` 往往在语义上没错，但整体交互体验仍然比应该有的更脆一些。

### 2. 命令面仍然会让用户太频繁地想着内部对象

`@current`、`@latest` 这类 selector 已经有明显帮助，但方向上还应该继续往“最自然的命令应该直接成立”去走，而不是频繁先查对象 id。

### 3. 各种读取面必须彼此一致

agent 的信心很依赖稳定的状态读取。

如果 `status`、`inspect` 和对象 `show` 之间哪怕短时间内打架，agent 就会开始花精力验证工具本身，而不是继续完成任务。对这些读取面的统一性，应该按产品关键路径来对待。

### 4. 对 agent 来说，真正的教材是 skill，不是 README

在 agent 使用场景里，真正的教程并不是仓库 README，而是 skill。

也就是说，产品表面不只是 CLI 自身，还包括：

- 命令行为
- 错误恢复
- 稳定的机器读取界面
- 教会 agent 正确循环方式的 skill

## 对后续开发的建议

### 优先级 1：先把 agent happy path 打磨顺，再扩展模型

在继续引入很多新对象或新概念之前，先把主循环磨顺：

- `inspect`
- 选择下一步
- 写一次
- 重新读状态
- 继续

### 优先级 2：把恢复路径当成功能本身，而不只是报错文案

一旦发生冲突，CLI 应该让恢复路径足够明确、足够低摩擦。

具体 next action、selector 和机器可读的 candidate 信息，远比“只是返回一个正确错误码”更有价值。

### 优先级 3：让 `inspect` 成为最可信的运行时事实来源

对 agent 工作流来说，一个绝对可信的状态面，比几个部分重叠的读取接口更重要。

`inspect --json` 应该继续朝“下一步动作的单一事实来源”去收敛。

### 优先级 4：把 skill 当成一等产品面来维护

如果希望 agent 真正把 Intent 用顺，skill 的维护严肃程度就应该接近命令文档本身。

workflow discipline、selector 用法、恢复预期和 read-before-write 模式，都是在 skill 里真正变成可执行经验的。

### 优先级 5：继续做 clean machine 与真实 agent 的端到端验证

现在 repo-backed install 和 skill 安装路径已经是用户旅程的一部分。

所以端到端验证要持续盯住这些点：

- clean machine 安装
- PATH 暴露
- 真实 agent 检测
- skill 放置位置
- 安装后的首次可用性

## 一句话总结

Intent 已经越过了“有趣原型”的阶段，进入了“有实际价值的工作层”。

下一步不应该急着横向扩张，而应该先把核心 agent 工作流打磨到足够稳定，让这层语义系统在日常使用里变得自然。
