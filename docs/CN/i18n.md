# Intent 文档国际化方案

目的：定义 Intent 文档如何演进为一套结构对称的 EN/CN 双语文档。

这份文档回答：

- 双语文件应该如何组织
- `EN/CN` 命名规则应该如何落地
- EN/CN 文件之间应如何同步
- 哪些文件应该先迁移

这份文档不回答：

- CLI contract 细节
- 实现路线图细节
- 发布或社区运营问题

## 目标

- 为核心文档同时提供 EN 和 CN 版本
- 避免双语内容长期漂移
- 控制维护成本

## 当前状态

- 仓库正处于双语迁移过程中
- GitHub 入口现在优先展示英文 README
- 对外入口文档已经具备 EN/CN 成对版本
- `cli` 和 `roadmap` 也已经具备 EN/CN 成对版本

## 核心原则

- 使用对称的 EN/CN 命名
- 避免在同一个文件里做中英混排
- 使用 `README.md` 作为英文 GitHub 入口，`README.CN.md` 作为中文根目录版本

## 目录策略

目标结构：

```text
README.md
README.CN.md

docs/
  README.md
  EN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
  CN/
    README.md
    glossary.md
    vision.md
    cli.md
    demo.md
    roadmap.md
    i18n.md
```

规则：

- `README.md` 是英文 GitHub 展示入口
- `docs/README.md` 是一个轻量的语言选择入口
- 文档主体放在 `docs/EN/` 和 `docs/CN/` 两棵语言树下
- 两棵语言树中的文件名保持一致
- 根目录 README 仍是唯一命名例外：英文使用 `README.md`，中文使用 `README.CN.md`

## 命名规则

- 英文文档使用 `docs/EN/`
- 中文文档使用 `docs/CN/`
- 两棵语言树中的文件名保持一致

示例：

- `docs/EN/vision.md` <-> `docs/CN/vision.md`
- `docs/EN/cli.md` <-> `docs/CN/cli.md`
- `README.md` <-> `README.CN.md`

## 同步规则

- 每对文档都应该在顶部直接链接到对方
- 如果一次修改先落某一种语言，另一种语言也应被明确同步，而不是长期放任漂移
- 必要时可以加入 `Last synced with: <commit>` 说明同步点

## 链接规则

- `README.md` 应链接 `README.CN.md`
- 英文文档默认只链接英文文档
- 中文文档默认只链接中文文档
- 顶部语言切换链接负责跨语言跳转
- 文档索引应清楚暴露两种语言路径

## 编写规则

- CLI 命令、对象 id、flags、JSON 字段名不翻译
- 核心对象术语保持英文：`intent`、`checkpoint`、`adoption`、`decision`、`run`
- 英文写作应以自然英文重写，而不是逐句直译
- 两种语言都采用清晰、中性的开源项目文档语气
- 避免在同一篇文档中使用双语段落

## 迁移顺序

Phase 1：对外入口文档

- `README.md`
- `README.CN.md`
- `docs/EN/README.md`
- `docs/CN/README.md`
- `docs/EN/vision.md`
- `docs/CN/vision.md`
- `docs/EN/demo.md`
- `docs/CN/demo.md`
- `docs/EN/glossary.md`
- `docs/CN/glossary.md`

Phase 2：项目 contract 文档

- `docs/EN/cli.md`
- `docs/CN/cli.md`
- `docs/EN/roadmap.md`
- `docs/CN/roadmap.md`

Phase 3：补齐双语覆盖

- `docs/EN/i18n.md`
- `docs/CN/i18n.md`
- 后续新增文档尽量直接以 EN/CN 成对方式创建

## 推荐执行方式

Step 1：

- 保留 `README.md` 作为英文 GitHub 展示入口
- 将 `README.CN.md` 作为中文根目录版本

Step 2：

- 把对外文档迁移到 `docs/EN/` 和 `docs/CN/`
- 更新索引和交叉链接

Step 3：

- 为目前只有中文的 `cli`、`roadmap` 补上英文版
- 在未补齐前，保持链接显式可见

Step 4：

- 验证 GitHub 入口是否英文优先
- 验证双语切换是否仍然足够直接

## 阶段完成标准

Phase 1 完成时：

- `README.md` 已经作为英文根目录 README
- `README.CN.md` 已存在
- 对外入口文档已经形成 EN/CN 成对版本

Phase 2 完成时：

- `docs/EN/cli.md` / `docs/CN/cli.md` 都已经存在
- `docs/EN/roadmap.md` / `docs/CN/roadmap.md` 都已经存在
- 主要文档链接不再依赖旧的不带语言后缀命名

Phase 3 完成时：

- 核心文档都具备 EN/CN 成对版本
- 成对链接和同步说明在整个文档集中保持一致

## 维护策略

- 新的公开文档应尽量直接以 EN/CN 成对形式创建
- 如果某一种语言先落地，另一种语言要被显式排入同步计划

## 简短结论

Intent 应保持 `README.md` / `README.CN.md` 的根目录结构，并将 `docs/` 组织成镜像的 `EN/` 与 `CN/` 语言树。
